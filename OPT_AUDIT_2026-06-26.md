## Fixed in this pass (branch `optaudit-jun26`)

### High
- **Daily update window could make auto-updates silently never run.** The window was enforced only as a runtime *skip* (`return Result.success()`), but the periodic work fires at a fixed phase independent of the window — so for long intervals (24h/48h/weekly) or narrow windows the phase could permanently miss the window. **Fix:** on an out-of-window skip, both `MangaLibraryUpdateJob`/`AnimeLibraryUpdateJob` now enqueue a one-time **catch-up** worker (`WORK_NAME_AUTO_CATCHUP`, `ExistingWorkPolicy.REPLACE`) with `setInitialDelay` to the next window start, computed by the new pure helper `LibraryPreferences.millisUntilNextWindowStart(...)`; the catch-up carries `WORK_NAME_AUTO` so the window/manual guards still apply (in-window → update; still-out → reschedules). Window prefs in `SettingsLibraryScreen` now `onValueChanged → setupTask(...)` so changes re-align the schedule.

### Medium
- **Optimistic chapter swap not rolled back on failed source switch** (`MangaScreenModel.switchSource`). Reordered so `freezeQueuedPageLists` + `setMangaSource` (server switch) succeed **before** the chapter list is committed to the DB — a failed switch now leaves both stored chapters and the effective source untouched (no chapter↔page mismatch).
- **No re-entry guard on `switchSource`** → rapid switches raced and could persist the earlier tap. Added a tracked `switchSourceJob` that cancels the prior switch so the latest wins; cancellation is rethrown at every catch (incl. the inner per-source-read catch) so a cancelled switch never proceeds to `setMangaSource`.
- **Manga download-queue restore blocked the main thread** (`MangaDownloader.init`). Moved off `launchNow`/Main onto the IO `scope`, matching the anime sibling.

### Low
- **Always-on log buffer captured all priorities**, letting DEBUG spam evict the errors the feature exists to keep. `InMemoryLogcatBuffer` now has a `minPriority` threshold (default INFO; App raises it to VERBOSE when verbose logging is enabled).
- **Notification channels raced early notifications** — the incognito-mode flow could `notify` before channels existed. The incognito collector now starts inside the same coroutine, after `setupNotificationChannels()`.
- **Lazy/invalidated multi-source prefetch.** Prefetch no longer fires eagerly on every multi-source screen open; it warms on first source-picker open (`warmSourcePrefetch()`, threaded via a new `onPickerOpened` callback) and the cache is cleared on manual refresh so switches don't show stale chapters.
- **Anime download-success check** compared `extension == ".tmp"` (with dot), which never matched (`UniFile.extension` has no dot) → stray temp files miscounted. Now `"tmp"`, matching the manga path.
- **Bulk manga-type set swallowed DB failures** — now captures the result and logs on failure (the library screen models have no user-facing error channel; logging matches sibling bulk actions).
- **`freezeQueuedPageLists` unbounded fan-out** capped at `limitedParallelism(3)`.
- Window KDoc corrected to "start-inclusive, end-exclusive" (matches the unchanged logic).
- Micro: `setMangaCategories`/`setAnimeCategories` `.map`→`.forEach` (drop throwaway list); `MangaTypeDetector` fused `.map{}.any{}` into one short-circuiting pass; `DebugLogsScreen.shareLogs` moved file I/O + debug-info gathering off the main thread.

## Follow-up pass (v0.18.1.21) — the two larger efficiency items

- **PERF F5 — Manhwa tab ran a second full library pipeline** (the Manga + Manhwa tabs are separate screen-model instances, each running the identical full-library `getLibraryFlow()` incl. per-entry download-count work). **Fixed:** the full-library flow is now built once and shared across both instances via a companion-cached `shareIn` (`SharingStarted.WhileSubscribed(stopTimeoutMillis = 5000, replayExpirationMillis = 0)`, `replay = 1`) on a singleton scope, capturing only app-singleton deps (no instance/state retention). The per-tab `MangaType` split stays downstream in `applyFilters`, so the two tabs still render correctly differentiated data from the one shared upstream; the cached map is released shortly after both tabs close.
- **PERF F9 — `getCommonCategories` N+1** (one category query per selected entry). **Fixed:** added a batched `getVisibleCategoriesBy{Manga,Anime}Ids(...)` query + repository/interactor returning `Map<id, List<Category>>`; the change-category dialog now issues a single query and groups in memory (selection order + zero-category entries preserved). Manga + anime.

## Deliberately NOT done — with evidence

- **`verifyMigrations` not enabled** in `data/build.gradle.kts`. Verified empirically: running `:data:verifyReleaseAnimeDatabaseMigration` **fails** on the pre-existing 2023 upstream migration `122.sqm` (the `animelibView` `LEFT JOIN(...)` view) — `(6,13): ')', '*' or <expr> expected, got 'E'`. The SQLDelight verifier's parser rejects migration SQL that runs fine on real SQLite (and has, on every install since 2023). Enabling the flag would block all CI builds; fixing it means rewriting historical migrations that have already executed on every device — high risk, no user benefit. Kept disabled.
